# Workflow: Execution

> Status: Authoritative

This workflow describes how the PM runs the job once field work is underway and the sequencing plan is being exercised.

---

## Outcome

By the end of the execution cycle for a gate:

- the active mobilizations and action items reflect what is happening on site
- blockers and follow-up work are visible
- gate classification changes remain deliberate and documented
- the shared system still reflects backend truth instead of drifting into local-only project stories

---

## Steps

1. Work from the current gate classification and change it explicitly when needed.
   Gates are execution groupings for mobilizations, not app-local guesses.

2. Work from the active mobilizations on the sequencer.
   Update status, timing, and reasoning as the real job evolves.

3. Use action items to track follow-up work and checkpoint items.
   Action items hold work that should stay visible even when it is broader than a single modal conversation or field note.

4. Capture field issues early.
   If execution reveals an RFI, change-order condition, or documentation need, keep the project record trail coherent even when part of the admin UI is still scaffolded.

5. Reclassify into the next gate deliberately.
   Do not move the project forward in the backend until the work associated with the current gate has actually advanced.

6. Hand off into the next gate with sequencing updates already reflected.
   The next gate should begin with an updated board, not with stale assumptions from the previous phase.

---

## Done When

- the shared system still matches the jobsite reality
- gate classification changes and sequencing changes can be explained after the fact
- the app reflects real project state rather than inventing a cleaner story than the backend supports
